[PickQ 발표 대본 - 14장]
발표 길이: 12~15분
사용법: 각 장에서 3~5문장 중 핵심 3문장만 선택해도 진행 가능

1장. 타이틀
- 안녕하세요. 저희는 스포츠 용품 역경매 플랫폼 PickQ를 개발한 팀입니다.
- PickQ는 구매자가 먼저 요청을 올리고, 판매자가 입찰로 제안하는 구조의 웹 서비스입니다.
- 이번 발표에서는 문제 정의부터 아키텍처, DB 설계, 실시간 기능, 그리고 구현 결과까지 전 과정을 보여드리겠습니다.
- 마지막에는 실제 데모 흐름과 채용 관점에서의 기술 강점까지 정리해 말씀드리겠습니다.

2장. 문제 정의와 기획 배경
- 기존 중고거래는 판매자가 올린 상품에서 구매자가 고르는 방식이라, 구매자 요구 조건 반영이 어렵다는 한계가 있었습니다.
- 특히 스포츠 용품은 종목, 상태, 예산, 브랜드 선호가 뚜렷해서 구매자가 원하는 조건 중심의 거래가 더 적합하다고 판단했습니다.
- 그래서 저희는 거래의 시작점을 판매자가 아니라 구매자 요청으로 바꾸는 역경매 모델을 선택했습니다.
- 이 선택이 이후 도메인 모델과 상태 전이 설계의 핵심 기준이 되었습니다.

3장. 서비스 가치 제안
- PickQ의 핵심 가치는 한 문장으로, 구매자가 조건을 제시하고 판매자가 맞춤 제안을 하는 거래 경험입니다.
- 첫째, 구매자는 원하는 조건에 맞는 제안을 비교해 선택할 수 있습니다.
- 둘째, 판매자는 필요한 고객에게 집중적으로 제안을 보낼 수 있어 거래 효율이 올라갑니다.
- 셋째, 거래 후 리뷰와 평판이 연결되어 다음 거래의 신뢰도를 높입니다.

4장. 주요 기능 요약
- 서비스는 회원, 역경매, 입찰, 거래, 커뮤니티, 실시간 기능으로 구성되어 있습니다.
- 사용자는 회원가입과 로그인 이후 경매 요청을 등록하고, 다른 사용자는 입찰을 통해 제안합니다.
- 낙찰 이후에는 주문, 결제, 배송, 구매확정, 리뷰로 이어지는 완결형 흐름이 동작합니다.
- 여기에 채팅과 알림을 실시간으로 연결해 사용자 반응성을 높였습니다.

5장. 사용자 플로우
- End-to-End 플로우는 구매요청 등록에서 시작해 입찰 수집, 낙찰자 선정, 거래 완료, 리뷰 작성으로 마무리됩니다.
- 구매자 관점에서는 조건 정의와 낙찰 선택이 핵심이고, 판매자 관점에서는 제안 품질과 빠른 커뮤니케이션이 핵심입니다.
- 저희는 이 두 흐름을 하나의 상태 기반 모델로 통합해 화면과 데이터가 일관되게 움직이도록 설계했습니다.
- 그래서 기능이 많아도 사용자 입장에서는 자연스럽게 한 줄의 거래 여정을 따라가게 됩니다.

6장. 기술 스택과 개발 환경
- 백엔드는 Java 21, Spring Boot 3.5, MyBatis, MySQL 조합으로 구성했습니다.
- 프론트는 Thymeleaf 기반 SSR에 Bootstrap과 JavaScript를 결합해 빠르게 화면을 구현했습니다.
- 실시간 기능은 Spring WebSocket과 STOMP, SockJS를 사용해 채팅과 알림을 처리했습니다.
- 협업은 Git/GitHub, 이슈 관리 도구, 디자인 도구를 사용해 스프린트 단위로 진행했습니다.

7장. 시스템 아키텍처
- 구조는 브라우저 요청이 컨트롤러, 서비스, MyBatis 매퍼를 거쳐 DB에 반영되는 전형적인 MVC 패턴입니다.
- 동시에 WebSocket 채널을 분리해 HTTP 요청과 실시간 이벤트가 서로 간섭 없이 동작하도록 했습니다.
- 즉, CRUD 중심 기능은 안정적으로 처리하고, 즉시성이 필요한 기능은 STOMP 토픽으로 브로드캐스트합니다.
- 이 분리 덕분에 유지보수성과 사용자 경험을 함께 확보할 수 있었습니다.

8장. DB 설계 철학
- DB는 코드 테이블 분리와 강한 제약조건 적용을 핵심 원칙으로 설계했습니다.
- role, grade, 상태 코드, 카테고리 코드를 분리해 운영 데이터의 일관성을 확보했습니다.
- 또한 UNIQUE, CHECK, FK, ON DELETE 정책을 기능 의도에 맞게 다르게 적용했습니다.
- 결과적으로 데이터 무결성과 확장성, 그리고 서비스 정책 반영의 균형을 맞췄습니다.

9장. 핵심 도메인 모델
- 핵심 도메인은 auction, bid, orders, payment, delivery, review가 상태 전이로 연결되는 구조입니다.
- 예를 들어 낙찰 시점에는 단순 상태 변경이 아니라 주문 생성과 이후 거래 단계로의 진입이 함께 일어납니다.
- 이때 bid 단위 unique 제약과 주문 상태 제약을 통해 중복 거래나 비정상 상태를 방지했습니다.
- 즉, 화면 기능이 아니라 도메인 규칙 중심으로 거래 라이프사이클을 통제했습니다.

10장. API 설계 규모
- Controller 전체 엔드포인트를 기능군별로 정리해 누락 없이 관리했습니다.
- 회원, 경매, 입찰, 주문, 결제, 리뷰, 알림, 채팅까지 REST 경로를 체계화했습니다.
- 여기에 HTTP 외에 STOMP 매핑도 명시해 실시간 경로까지 문서화했습니다.
- 문서화 덕분에 팀 내 역할 분담과 통합 테스트 효율이 크게 올라갔습니다.

11장. 실시간 기능 구현
- 채팅은 app 경로로 수신한 메시지를 chatroom 단위 topic으로 브로드캐스트하는 구조입니다.
- 알림은 사용자 식별자 기준 topic으로 전송해 개인화된 이벤트를 즉시 전달합니다.
- 즉, 같은 실시간 기술을 쓰더라도 기능별 라우팅 전략을 분리해 운영했습니다.
- 그 결과 사용자 입장에서 중요한 상태 변화가 페이지 새로고침 없이 반영됩니다.

12장. 운영 안정성 및 보안 대응
- 데모 프로젝트지만 운영 리스크를 고려해 로그인 실패 횟수 제한과 계정 잠금 로직을 적용했습니다.
- 세션 기반 인증 흐름과 인터셉터 리다이렉트 처리를 통해 비정상 접근을 줄였습니다.
- 또한 커스텀 에러 페이지와 예외 노출 제한 설정으로 장애 시 사용자 영향도를 낮췄습니다.
- 데이터는 soft delete 전략을 함께 사용해 복구 가능성과 이력 보존을 고려했습니다.

13장. 성과와 회고
- 저희 팀의 성과는 기능 개수보다 복잡한 상태 기반 도메인을 끝까지 통합했다는 점에 있습니다.
- 특히 역경매-입찰-거래-리뷰 흐름을 실시간 기능과 결합해 사용자 경험을 완성한 것이 핵심입니다.
- 구현 과정에서 병합 충돌, 상태 전이 예외, 연동 이슈를 반복적으로 해결하며 협업 품질도 높였습니다.
- 이 프로젝트를 통해 설계 선택이 코드 품질과 운영 안정성에 직접 연결된다는 점을 체감했습니다.

14장. 채용 어필 포인트 및 마무리
- 저희의 가장 큰 강점은 CRUD를 넘어서 상태 전이형 거래 도메인을 구현했다는 점입니다.
- 두 번째는 WebSocket/STOMP 기반 실시간 채팅과 알림을 실제 서비스 흐름에 통합한 경험입니다.
- 세 번째는 DB 무결성 제약과 협업 프로세스를 함께 가져가며 실무형 개발을 수행했다는 점입니다.
- 이상으로 발표를 마치며, 데모 시연과 질의응답으로 이어가겠습니다. 감사합니다.
