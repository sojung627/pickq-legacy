[PickQ 2인 실전 시연 시나리오 - 역경매 End-to-End 1사이클]
목표: 짧고 강하게, 핵심 로직과 실시간성, 모바일 최적화를 동시에 증명
총 소요: 7분 30초 ~ 9분
권장 인원: 2인(필수) + 1인(선택, 진행 보조/타이머)

=================================
1) 시연 콘셉트(한 줄)
=================================
"구매자가 요청을 만들고, 판매자가 제안하고, 실시간으로 협의한 뒤, 낙찰-거래완료-리뷰까지 한 번에 닫는다."

=================================
2) 역할 분담
=================================
- 진행자 A(구매자 계정)
: 전체 흐름 리딩, 경매 생성/낙찰/구매확정/리뷰 담당
- 진행자 B(판매자 계정)
: 입찰 등록/채팅 응답/배송 처리 담당
- 선택 보조 C(있으면 좋음)
: 타이머, 다음 화면 큐, 장애 시 플랜B 호출

=================================
3) 화면 배치(집중도 유지 핵심)
=================================
- 왼쪽 대형 화면: A(데스크톱 브라우저)
- 오른쪽 대형 화면: B(모바일 뷰 또는 실제 스마트폰 미러링)
- 원칙
1. A가 설명할 때 B는 조작 대기
2. 상태 변화가 발생하면 반드시 반대편 화면에서 즉시 결과를 확인
3. "입력"보다 "변화 결과"를 오래 보여준다

=================================
4) 사전 세팅(시작 10분 전)
=================================
- 테스트 계정 2개 준비
: buyer_demo / seller_demo
- 샘플 데이터 정리
: 카테고리/회원/주소/기본 이미지 정상 여부 확인
- 소켓 연결 확인
: 채팅/알림 아이콘이 정상 동작하는 화면 미리 점검
- 결제 데모 방식 합의
: 실결제 대신 성공 흐름 중심으로 시연할지 사전 결정
- 모바일 반응형 확인 포인트 사전 지정
: 메뉴, 카드, 상세페이지, 채팅창, 마이페이지 레이아웃

=================================
5) 본 시연 타임라인(권장 8분)
=================================

[0:00~0:30] 오프닝
- A 멘트
: "지금부터 역경매 핵심 사이클을 8분 안에 끝까지 보여드리겠습니다."
: "왼쪽은 구매자, 오른쪽은 판매자 모바일 화면입니다."

[0:30~1:40] 구매 요청 생성(A)
- A 동작
1. 로그인
2. 경매 등록 페이지 진입
3. 카테고리/희망가/설명/마감 입력 후 등록
- 보여줄 핵심
: 생성 직후 목록/상세에 새 경매가 노출
: 상태가 진행중(open)으로 시작
- 짧은 멘트
: "거래의 시작점이 판매글이 아니라 구매요청이라는 것이 핵심입니다."

[1:40~2:40] 판매자 입찰(B, 모바일 중심)
- B 동작
1. 모바일에서 동일 경매 진입
2. 입찰 등록(상품 이미지 + 가격 + 메시지)
- 보여줄 핵심
: 모바일에서도 폼/버튼/이미지 업로드 동선이 깨지지 않음
: 제출 후 A 화면 상세 입찰 목록에 반영
- 짧은 멘트
: "반응형 최적화로 모바일에서도 거래 핵심 액션을 동일하게 수행합니다."

[2:40~3:40] 실시간 채팅/알림 확인(A<->B)
- A가 채팅 1건 전송, B가 즉시 답장
- 양쪽 모두 알림/읽지않음 변화 확인
- 보여줄 핵심
: 새로고침 없이 메시지/알림 즉시 반영
: 실시간 경로 분리(채팅 topic, 알림 topic)
- 짧은 멘트
: "중요 상태 변경은 REST, 즉시 상호작용은 WebSocket으로 처리합니다."

[3:40~4:40] 낙찰 선택(A)
- A 동작
1. 입찰 상세에서 낙찰자 선정
2. 주문 내역 화면으로 이동
- 보여줄 핵심
: 낙찰과 동시에 거래 흐름으로 진입
: 경매-입찰-주문 연결이 끊기지 않음
- 짧은 멘트
: "낙찰은 단순 상태 변경이 아니라 주문 라이프사이클 시작 이벤트입니다."

[4:40~6:10] 거래 상태 전이(B -> A)
- B 동작
1. 배송 처리(운송장/발송 처리)
- A 동작
2. 구매자 화면에서 구매확정
- 보여줄 핵심
: 주문 상태가 단계적으로 변경됨
: 판매자/구매자 화면이 같은 상태를 일관되게 표시
- 짧은 멘트
: "orders-payment-delivery-confirm 흐름으로 엔드투엔드가 닫힙니다."

[6:10~7:00] 리뷰 작성 및 평판 반영(A)
- A 동작
1. 리뷰 작성
2. 프로필/리뷰 조회에서 결과 확인
- 보여줄 핵심
: 거래 결과가 평판 데이터로 누적
- 짧은 멘트
: "거래 종료 데이터가 다음 거래 신뢰도로 이어집니다."

[7:00~8:00] 마무리 요약
- A 멘트(정리 3줄)
1. "구매요청 기반 역경매 사이클을 끝까지 시연했습니다."
2. "모바일 반응형과 실시간 웹소켓 상호작용을 동시에 검증했습니다."
3. "상태 전이형 거래 로직을 사용자 경험으로 연결한 것이 핵심 성과입니다."

=================================
6) 절대 늘어지지 않게 하는 운영 규칙
=================================
- 규칙 1: 입력 화면 체류 10초 초과 금지
- 규칙 2: 설명은 액션 직전 1문장, 결과 직후 1문장만
- 규칙 3: 같은 화면에서 3클릭 이상 필요하면 사전 데이터로 점프
- 규칙 4: 장애가 나면 15초 안에 플랜B 전환

=================================
7) 플랜B(문제 발생 시 즉시 전환)
=================================
- 결제 외부 연동 지연
: 결제 성공/실패 페이지 라우팅만 확인 후 주문 상태 화면으로 바로 이동
- 웹소켓 지연
: 채팅/알림 REST 조회 화면으로 즉시 대체해 데이터 반영성 설명
- 데이터 꼬임
: 사전 준비한 데모 경매번호로 바로 진입

=================================
8) 심사위원/면접관 집중 포인트(질문 유도)
=================================
- "왜 역경매 모델을 선택했는가?"
: 구매자 요구 중심 거래를 위해
- "실시간 처리를 왜 분리했는가?"
: CRUD 안정성과 즉시성 기능을 분리해 유지보수성 확보
- "핵심 기술 난이도는 무엇인가?"
: 상태 전이 일관성 + 사용자 양측 화면 동기화

=================================
9) 리허설 체크리스트(최소 2회)
=================================
- 1차: 시간 체크(8분 초과 여부)
- 2차: 장애 상황 강제 발생 후 플랜B 동작 확인
- 최종: 멘트 축약본으로 6분 버전도 준비

[한 줄 결론]
짧은 데모의 핵심은 "입력 과정"이 아니라 "양측 화면에서 즉시 확인되는 상태 변화"를 보여주는 것이다.
