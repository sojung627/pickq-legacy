[PickQ 라이브 데모 시나리오]
총 6~8분 권장

0. 데모 시작 전 체크리스트(사전 준비)
- 테스트 계정 2개 준비: 구매자 1, 판매자 1
- 초기 데이터: 카테고리/회원/샘플 경매 존재 확인
- 브라우저 2개 세션 준비(또는 시크릿 창 분리)
- 채팅/알림 소켓 연결 확인

1. 시나리오 목표 한 줄 소개 (20초)
- 멘트: "구매 요청 생성부터 입찰, 낙찰, 거래, 리뷰까지 한 번에 보여드리겠습니다."

2. 구매자 플로우 (2분)
- 로그인 -> 경매 등록(/auctions/new)
- 카테고리/희망가/마감일/설명 입력 후 등록
- 생성된 경매 상세 진입
- 강조 포인트: 상태가 open으로 생성되고 목록/상세에 즉시 반영

3. 판매자 플로우 (2분)
- 판매자 계정 로그인
- 해당 경매에 입찰 등록(/auctions/{auctionIdx}/bids)
- 상품 이미지 + 제안가 입력 후 제출
- 강조 포인트: 입찰 데이터가 경매 상세에 누적, 구매자에게 알림 발생

4. 실시간 채팅/알림 (1분)
- 구매자/판매자 화면을 나란히 시연
- 채팅 전송 후 상대 화면 즉시 반영 확인
- 최근 알림/읽지 않음 카운트 확인
- 강조 포인트: STOMP topic 기반 실시간 UX

5. 낙찰 및 거래 진행 (1~2분)
- 구매자 계정에서 낙찰자 선정(/auctions/{auctionIdx}/bids/{bidIdx}/win)
- 주문 내역(/mypage/orders)에서 상태 확인
- 결제 성공/배송 처리/구매확정 시나리오를 빠르게 이동 시연
- 강조 포인트: orders-payment-delivery 상태 전이

6. 리뷰 작성/프로필 반영 (40초)
- 거래 완료 후 리뷰 작성
- 프로필 또는 리뷰 조회 페이지에서 반영 확인
- 강조 포인트: 거래 종료 데이터가 평판으로 연결

7. 마무리 멘트 (20초)
- 멘트: "역경매 기반 거래의 전 주기를 실시간 경험까지 포함해 구현했습니다."

[데모 중 어필해야 할 기술 포인트]
- 상태 코드 분리(auction_status, bid_status, order_status)
- DB 제약조건(UNIQUE, CHECK, FK)으로 무결성 확보
- 세션/인터셉터 기반 인증 흐름
- WebSocket + STOMP 경로 분리

[장애 대비 플랜 B]
- 결제 외부 연동 실패 시: 결제 성공/실패 페이지 라우팅만 설명하고 주문 상태 화면으로 전환
- 소켓 이슈 시: REST API 채팅/알림 조회 화면으로 대체 시연
- 데이터 꼬임 시: 준비한 샘플 경매 번호로 바로 이동

[질의응답 예상 질문]
1) 왜 JPA 대신 MyBatis를 선택했나?
- SQL 제어 가시성, 복잡 쿼리 튜닝, 학습 목표 적합성을 강조.
2) 트랜잭션 경계는 어떻게 관리했나?
- 낙찰-주문생성-상태변경을 서비스 계층에서 묶어 일관성 유지.
3) 확장 시 우선 개선 포인트는?
- notification FK 제약 활성화, 결제/배송 상태 이벤트 분리, 테스트 자동화 강화.
